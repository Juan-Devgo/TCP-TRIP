# ADR-005: Internacionalización mediante file-based routing de Astro con translations.ts

**Estado:** Aceptado  
**Fecha:** 2026-04-14  
**Última actualización:** 2026-04-14 (D-01: español por defecto validado)  
**Deciders:** Equipo TCP-TRIP  
**Motiva:** UX-001, T-001, T-002, UX-001

---

## Contexto

TCP-TRIP es una plataforma bilingüe español/inglés (UX-001). La cobertura bilingüe completa es un criterio de evaluación académica (PR-003, objetivo de producto #5). Todas las cadenas de texto deben estar internacionalizadas; ningún texto puede estar hardcodeado en los componentes.

Las opciones para gestionar i18n en Astro son:

**Opción A: File-based routing de Astro** — Cada ruta existe dos veces: `/tcp-ip/` (inglés) y `/es/tcp-ip/` (español). El idioma se detecta desde la URL. Las cadenas se almacenan en un objeto TypeScript centralizado (`translations.ts`).

**Opción B: i18next + react-i18next** — Librería i18n popular en el ecosistema React. Gestiona las traducciones con archivos JSON por idioma y detecta el idioma via cookies o headers.

**Opción C: Astro i18n integration (routing nativo)** — Astro 4+ incluye soporte de i18n nativo en `astro.config.mjs` con `i18n: { locales, defaultLocale }`. Complementa el file-based routing.

La decisión ya está implementada (el código base usa file-based routing con `translations.ts` y `astro.config.mjs` con `i18n`). Este ADR documenta la justificación y las convenciones.

---

## Decisión

Se usa **file-based routing de Astro** para el enrutamiento por idioma, combinado con un **objeto de traducciones centralizado** en `src/i18n/translations.ts`.

- Las rutas en **español** son las canónicas y se sirven bajo la raíz del sitio (e.g., `/tcp-ip/AppLayer.astro`, `/protocol-creator/`).
- Las rutas en **inglés** son la variante alternativa y se sirven bajo el prefijo `/en/` (e.g., `/en/tcp-ip/AppLayer.astro`, `/en/protocol-creator/`).
- El idioma activo se determina leyendo si la URL comienza con `/en/`; en caso contrario, el idioma es español.
- Las cadenas de texto se obtienen via `t('namespace.key')` donde `t` es una función helper creada a partir del idioma detectado.
- La configuración de Astro (`astro.config.mjs`) usa `i18n: { locales: ['es', 'en'], defaultLocale: 'es' }` para el enrutamiento automático.

**Resolución del conflicto de idioma por defecto (D-01 — validado 2026-04-14):**

El equipo ha validado que el **español es el idioma canónico** por las siguientes razones:

1. El producto está dirigido primariamente a universidades colombianas de habla hispana.
2. El criterio de evaluación académica (PR-003) se aplica en el contexto de la Universidad del Quindío, cuya lengua institucional es el español.
3. Los usuarios primarios (estudiantes y docentes de redes en Colombia) esperarán el contenido en español como experiencia de primera carga.

**Cambios requeridos en el código (acción para el desarrollador):**

| Archivo | Cambio |
|---------|--------|
| `astro.config.mjs` | Cambiar `defaultLocale: 'en'` a `defaultLocale: 'es'` |
| `src/i18n/translations.ts` | Confirmar que `defaultLang = 'es'` (ya está correcto) |
| `src/pages/` | Las rutas en español permanecen aquí (son la raíz canónica) |
| `src/pages/es/` | Renombrar a `src/pages/en/` — estas son las rutas alternativas en inglés |
| Middleware | Actualizar `createRouteMatcher` para que las rutas de idioma alternativo usen `/en/` en lugar de `/es/` |
| Layouts | `Layout.astro` pasa a ser el layout en español; crear `en/Layout.astro` para inglés |

**Impacto en la estructura de rutas:**

```
/               → landing en español (canónica)
/tcp-ip/        → explorador TCP/IP en español (canónica)
/protocol-creator/ → constructor de protocolos en español
/en/            → landing en inglés (alternativa)
/en/tcp-ip/     → explorador TCP/IP en inglés
/en/protocol-creator/ → constructor de protocolos en inglés
```

**Nota importante:** La estructura actual del repositorio tiene las rutas españolas bajo `/es/` y las inglesas bajo la raíz. Este ADR documenta la estructura objetivo. La migración física de archivos se realiza como parte de P-001 (ver `docs/architecture/directory-structure.md`).

## Consecuencias

### Positivas

- **Sin dependencias adicionales:** El file-based routing es nativo de Astro. No se necesita instalar `i18next`, `react-i18next`, `next-intl` ni ninguna librería externa.
- **TypeScript-first:** Las traducciones en `translations.ts` son un objeto TypeScript fuertemente tipado. El compilador detecta si se accede a una clave que no existe. Es más seguro que archivos JSON sin tipado.
- **URLs limpias y predecibles:** `/tcp-ip/` y `/es/tcp-ip/` son URLs independientes y SEO-friendly. No hay `/tcp-ip/?lang=es` ni cookies de idioma.
- **Sin complejidad de hidratación:** El idioma se pasa desde el servidor al island como prop. Los componentes React no necesitan detectar el idioma en el cliente ni configurar un provider de i18n.
- **Simple de auditar:** Para verificar cobertura i18n (T-001), basta con buscar strings hardcodeados en los archivos de componentes. No hay archivos de recursos JSON dispersos.

### Negativas / Trade-offs

- **Duplicación de archivos de página:** Cada nueva página se crea dos veces: una en `src/pages/` y otra en `src/pages/es/`. Para el equipo de 1-2 personas, esto es una carga de mantenimiento real. Si se agrega una nueva sección con 10 páginas, son 20 archivos `.astro` a crear. Mitigación: los archivos de página en español típicamente son muy similares al inglés (solo cambia el import del layout y la prop `lang`).
- **Sin cambio de idioma dinámico:** No hay un botón "cambiar idioma" que navegue automáticamente a la ruta equivalente en el otro idioma. El usuario debe conocer la URL `/es/` o navegar desde la landing. Implementar el cambio dinámico requiere lógica de mapeo de URLs que no está implementada aún.
- **Escalabilidad del objeto `translations.ts`:** A medida que crezca el número de secciones y cadenas, el objeto de traducciones crecerá. Para V2.0 (admin, perfil, modo clase), el archivo puede volverse grande. Si supera ~500 líneas, considerar dividirlo por namespace (`translations/nav.ts`, `translations/admin.ts`, etc.).
- **Cadenas en componentes React:** Los componentes React (islands) reciben el idioma como prop y necesitan acceder al objeto de traducciones. Si el componente es profundo en el árbol, pasar `lang` por props a todos los sub-componentes puede volverse verboso. Alternativa: un contexto React local por island (no global) para distribuir el idioma. Evaluar en V1.1.

### Riesgos

- **R-004 (del PO doc):** Textos hardcodeados descubiertos tarde en el ciclo de desarrollo. Mitigación: T-001 (auditoría de cobertura i18n) es obligatoria antes de cerrar V1.0.
- **Cobertura incompleta en componentes React:** Los 27+ sub-componentes de `ProtocolBuilder` pueden tener textos sin internacionalizar. Prioritario en T-001.

---

## Convenciones Establecidas

1. **Ningún texto hardcodeado en componentes.** Todo texto visible va en `translations.ts`. Sin excepciones.
2. **Namespace por sección:** Las claves se organizan como `sección.subsección.clave` (e.g., `admin.roleRequests.approveButton`).
3. **Agregar siempre las dos versiones (es + en) simultáneamente.** No hacer commit de una clave sin su traducción en ambos idiomas.
4. **El idioma se pasa como prop `lang: Language` a los islands React.** No usar `window.location` para detectar el idioma en el cliente.
5. **Toda nueva ruta se crea en inglés y español simultáneamente.** No dejar rutas sin su espejo `/es/`.

---

## Alternativas Evaluadas

| Alternativa | Razón de descarte |
|-------------|-------------------|
| i18next + react-i18next | Agrega una dependencia externa y un provider de React que envuelve todo el árbol de cada island. Innecesario cuando Astro ya provee los mecanismos de i18n nativos. |
| Archivos JSON por idioma (`es.json`, `en.json`) | Sin tipado TypeScript, el compilador no detecta claves faltantes o erróneas. El objeto TypeScript es superior para un proyecto que ya usa TypeScript estricto. |
| Un solo archivo de página con detección de idioma en runtime | Rompe el modelo SSR de Astro: las páginas SSR se compilan por ruta. Una página con lógica de idioma en runtime sería más compleja y menos eficiente que dos páginas simples. |
